Обнаружь изменения в проекте и точечно обнови вики: $ARGUMENTS

─────────────────────────────────────────
ЭТАП 0 — РЕГЕНЕРАЦИЯ REPO-MAP (автоматически)
─────────────────────────────────────────

```bash
python wiki/scripts/repo-map.py
```
> Агент выполняет это сам перед анализом изменений. Обновляет wiki/architecture/repo-map.md и wiki/architecture/file-page-map.json.

─────────────────────────────────────────
ЭТАП 1 — ОБНАРУЖЕНИЕ ИЗМЕНЕНИЙ
─────────────────────────────────────────

Если $ARGUMENTS не указан — обнаруживай изменения автоматически:

Шаг 1а. Попробуй git (предпочтительно):
  git diff --name-only HEAD          # несохранённые изменения
  git diff --name-only HEAD~1 HEAD   # последний коммит
  git status --porcelain             # новые неотслеживаемые файлы

  Из вывода отфильтруй только пути, которые входят в метки wiki/sources.md.
  Если git недоступен → перейди к шагу 1б.

Шаг 1б. Fallback по дате изменения файлов:
  Прочитай wiki/log.md или колонку `Индексировано` в wiki/sources.md — найди дату.
  Найди все файлы из путей wiki/sources.md, изменённые после этой даты.

  Linux/macOS:
    find {путь} -newer wiki/log.md -type f

  Windows (PowerShell):
    Get-ChildItem -Path {путь} -Recurse -File | Where-Object { $_.LastWriteTime -gt (Get-Date "{дата_индексации}") }

Шаг 1в. Если $ARGUMENTS указан — использовать как явный список файлов или путей.

Шаг 1г. SHA-256 верификация (точный детектор, если git недоступен):
  Прочитай `wiki/.source-hashes.json` — это словарь: `путь → sha256`.
  Для каждого файла из путей sources.md пересчитай хеш:
    Linux/macOS: sha256sum {файл}
    Windows: Get-FileHash {файл} -Algorithm SHA256
  Сравни с сохранённым: не совпадает → изменён; нет в словаре → новый.
  Работает точнее mtime — улавливает semantic-merge и переименования.

Шаг 1д. Загрузить маппинг файл → страница:
  Прочитай `wiki/architecture/file-page-map.json` — это точный словарь: исходный файл → wiki-страница.
  Для каждого изменённого файла ищи точное попадание в ключах маппинга.

─────────────────────────────────────────
ЭТАП 2 — КЛАССИФИКАЦИЯ ИЗМЕНЕНИЙ
─────────────────────────────────────────

Для каждого обнаруженного файла определи тип изменения:

НОВЫЙ ФАЙЛ (не упомянут ни в одной wiki-странице):
  → Тип: ДОБАВЛЕНИЕ
  → Действие: предложить /wiki-ingest для соответствующей метки

ИЗМЕНЁННЫЙ ФАЙЛ (есть упоминания в вики):
  Используй `file-page-map.json` — он даёт точное попадание: файл → wiki-страница.
  Если файл есть в маппинге → целевая страница известна точно.
  Если файла нет в маппинге → fallback: поиск по ключевым словам в wiki/modules/.

  Прочитай файл. Оцени масштаб изменения:
  a) ТОЧЕЧНОЕ — добавлен метод/поле/конфиг-ключ, исправлен баг,
     переименована переменная:
     → Найди конкретную секцию затронутой wiki-страницы (через маппинг)
     → Обнови только эту секцию
     → **Статус**: active (не меняется)
     → Не ставь [устарело] на всю страницу

  b) СТРУКТУРНОЕ — добавлена новая зависимость, изменена ответственность
     компонента, переименован класс/модуль, добавлен новый публичный API:
     → Обнови затронутые секции напрямую если изменение понятно
     → **Статус**: active (страница всё ещё актуальна, секции обновлены)
     → Пометь [изменено: {filename}] только если нужен человеческий взгляд

  c) ГЛОБАЛЬНОЕ — переработана архитектура, удалён компонент,
     слияние модулей:
     → **Статус**: stale (требует проверки человеком)
     → Пометить страницу [устарело — проверить по источнику]
     → Запустить каскадное обновление для связанных страниц

УДАЛЁННЫЙ ФАЙЛ:
  → Через `file-page-map.json` найди wiki-страницу, которой принадлежал файл
  → Добавить в секцию ## Tech debt: «[файл удалён — уточнить актуальность]»
  → Если файл был последним в модуле — пометить страницу модуля [устарело]

─────────────────────────────────────────
ЭТАП 3 — СВОДКА И ПОДТВЕРЖДЕНИЕ
─────────────────────────────────────────

До записи изменений покажи сводку:

  Обнаружено изменений: N файлов

  ДОБАВЛЕНИЯ (предложить ingestion):
    + src/новый-модуль/Service.java → не в file-page-map.json → предложить: /wiki-ingest src

  ТОЧЕЧНЫЕ (обновить секцию через file-page-map.json):
    ~ src/payment/PaymentService.java
      → wiki/modules/payment-service.md :: секция «Публичный API»
        + добавлен метод refundPayment(orderId)

  СТРУКТУРНЫЕ (обновить страницу через file-page-map.json):
    ~ src/auth/AuthFilter.java
      → wiki/modules/auth.md :: секции «Зависимости», «Конфигурация»
        + новая зависимость: ratelimit
        + новая переменная: AUTH_MAX_ATTEMPTS

  ГЛОБАЛЬНЫЕ (пометить [устарело]):
    ! src/core/ — переработан базовый слой
      → wiki/modules/core.md, wiki/architecture/module-map.md

  Применить изменения?

Дождись подтверждения пользователя. После подтверждения:
  - Выполнить все обновления (с проставлением **Статус**: active/stale)
  - Выполнить каскадное обновление для структурных и глобальных изменений
  - Обновить `wiki/.source-hashes.json` — пересчитать SHA-256 для всех затронутых файлов
  - Обновить `wiki/hot-context.md` — добавить строку об изменении
  - Добавить запись в wiki/log.md

─────────────────────────────────────────
ПРАВИЛА ПОМЕТОК
─────────────────────────────────────────

[изменено: filename]   — конкретный файл изменился, нужна проверка секции
[устарело]             — зарезервировано для глобальных структурных изменений
                         НЕ использовать для точечных правок

Ставить [устарело] на всю страницу только если:
  - удалён компонент которому посвящена страница
  - компонент слит с другим
  - кардинально изменилась ответственность модуля
