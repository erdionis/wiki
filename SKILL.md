# Навык: Wiki-координация проекта

## Назначение навыка

Этот навык описывает, как Agents ведёт вики для проекта вместо тяжёлого RAG.
Вики — единственный источник знаний при работе с проектом в новой сессии.

Навык подключается из `AGENTS.md` проекта и переиспользуется в любом проекте без изменений.

**Язык**: все страницы вики пишутся **на русском языке**.

---

## Основополагающая философия: wiki как единственный источник

**Wiki — это RAG, только проще.** Знания компилируются один раз и накапливаются.
Каждый вопрос задаётся вики, а не коду. Каждое обращение к коду заканчивается
записью в вики — чтобы второй раз не повторять тот же поиск.

Три правила, нарушать нельзя:

1. **Wiki first** — перед любой задачей или вопросом: сначала читать wiki.
   Не код, не память модели — wiki.
   Запуск: `/wiki-start <задача>` — восстанавливает контекст и сразу разворачивает
   wiki-first протокол для конкретной задачи.

2. **Code → Wiki** — нашёл что-то в коде, чего нет в wiki →
   записать в wiki немедленно, до продолжения работы.
   Второй раз читать тот же код не должно быть нужды.

3. **Never from memory** — Agents не отвечает из собственных знаний о проекте.
   Только из wiki. Если страницы нет — сказать явно и предложить ingest.

---

## Обязательный первый шаг в любой сессии

Перед ответом на любой вопрос о проекте или выполнением задачи:
1. Прочитать `wiki/sources.md` — понять, какие источники есть
2. Прочитать `wiki/architecture/overview.md` (если есть) — живой синтез проекта
3. **L1 hot cache**: если `wiki/hot-context.md` существует и обновлён < 24ч — прочитать его
   **L2 index**: если hot-context устарел или отсутствует — прочитать `wiki/index.md`
4. Прочитать релевантные страницы по задаче
5. Не читать исходный код повторно, если страница не помечена `[устарело]`

## Структура wiki/

```
wiki/
  sources.md        ← манифест: какие пути проекта считаются источниками
  index.md          ← оглавление всей вики
  log.md            ← append-only журнал операций
  architecture/     ← обзор системы, карта модулей, потоки данных
  modules/          ← по странице на каждый модуль / пакет / сервис
  concepts/         ← доменные понятия, паттерны, общий словарь
  decisions/        ← архитектурные решения (ADR)
  env/              ← конфиги окружений, переменные, секреты
  plans/            ← планы разработки, задачи, LLM-планы
  qa/               ← покрытие тестами, долг, известные баги
```

## Манифест источников (wiki/sources.md)

Единственное место, где указано что считается исходными данными.
Agents **никогда не изменяет** файлы из этого манифеста.
Обновляет манифест только человек.

Формат таблиц в `wiki/sources.md`:

```markdown
## Исходный код
| Метка       | Путь          | Описание              | Индексировано |
|-------------|---------------|-----------------------|---------------|
| `src-main`  | `src/`        | Основной код          | 2026-06-10    |
| `src-test`  | `tests/`      | Тесты                 | —             |

## Конфигурации
| Метка        | Путь        | Описание              | Индексировано |
|--------------|-------------|-----------------------|---------------|
| `config-app` | `config/`   | Конфигурация          | —             |
| `config-env` | `deploy/`   | Конфиги окружений     | —             |

## Документация
| Метка  | Путь        | Описание              | Индексировано |
|--------|-------------|-----------------------|---------------|
| `docs` | `docs/`     | Документация          | —             |
| `adr`  | `docs/adr/` | Архитектурные решения | —             |

## Планы
| Метка   | Путь      | Описание                          | Индексировано |
|---------|-----------|-----------------------------------|---------------|
| `plans` | `plans/`  | Планы разработки, LLM-задачи      | —             |

## Исключения — игнорировать
- .git/ .idea/ __pycache__/ target/ build/ dist/ *.pyc *.class *.jar wiki/
```

Колонка **Индексировано** — дата последней успешной индексации метки (проставляется автоматически при `/wiki-ingest`). Используется как fallback для mtime-based detection в `/wiki-update` когда git недоступен.

## Сценарии ingestion

Перед записью любой страницы применяется **дерево решений**:

```
Источник уже проиндексирован?
  ДА → существующая страница покрывает ту же суть?
         ДА  → слить новые данные в существующую страницу
                (добавить в Sources, обновить затронутые разделы)
         НЕТ → создать новую страницу рядом, добавить перекрёстные ссылки
  НЕТ → создать новую страницу
```

После создания или обновления основной страницы всегда выполняется **каскадное обновление**:

```
1. Проверить все страницы той же категории (modules/, concepts/ и т.д.)
   на пересечение содержимого с только что изменённой страницей
2. Просмотреть wiki/index.md — найти записи других разделов,
   связанных с изменёнными данными
3. Обновить каждую затронутую страницу; освежить дату «Последнее обновление»
4. Зафиксировать все затронутые страницы в записи wiki/log.md
```

### WIKI-INGEST-MODULE: Ingestion модуля / компонента

1. Прочитать `wiki/sources.md` — найти путь по метке
2. **Проверить `wiki/.ingest-progress.json`** — есть ли незавершённый прогресс для этой метки (не старше 24ч). Если да — спросить пользователя: продолжить с прерванного места?
3. Прочитать все файлы по этому пути
4. Определить: ответственность, публичный API, зависимости, побочные эффекты
5. **Применить дерево решений** (слить или создать новую страницу)
6. Если создаётся новая: `wiki/modules/{имя}.md`
7. Создать/обновить страницы в `wiki/concepts/` для новых доменных терминов
8. Добавить/обновить строку в `wiki/architecture/module-map.md`
9. **После создания/обновления КАЖДОЙ wiki-страницы:**
   - Обновить `wiki/architecture/file-page-map.json`: добавить запись `исходный_файл → wiki/страница.md`
   - Добавить обработанный файл в `wiki/.ingest-progress.json` → `processed_files`
   - Сохранить прогресс (phase, current_label, last_file, current_file_index)
10. **Выполнить каскадное обновление**
11. **Обсудить с пользователем ключевые находки и противоречия**
12. **В конце успешной индексации метки:**
   - Обновить колонку `Индексировано` в `wiki/sources.md` для текущей метки (текущая дата)
   - Создать/обновить `wiki/.source-hashes.json` — вычислить SHA-256 для каждого прочитанного файла
   - Очистить `wiki/.ingest-progress.json` (прогресс завершён)
   - Обновить `wiki/hot-context.md` — добавить информацию о новых страницах
   - Обновить `wiki/index.md`, добавить запись в `wiki/log.md`

### WIKI-INGEST-CONFIG: Ingestion конфигурации

1. Прочитать `wiki/sources.md` — найти метки `config-*`
2. **Проверить `wiki/.ingest-progress.json`** — продолжить прерванную индексацию при наличии
3. Извлечь: переменные окружения, feature-флаги, эндпоинты, структуру секретов
4. **Применить дерево решений** (слить или создать)
5. Создать/обновить `wiki/env/{среда}.md`
6. Явно зафиксировать отличия между средами; отметить противоречия между конфигами
7. **После каждой страницы:** обновить `file-page-map.json` (конфиг-файл → wiki/env/...), сохранить прогресс в `.ingest-progress.json`
8. **Выполнить каскадное обновление** (проверить modules/, environment.md)
9. **В конце:** обновить `Индексировано` в `sources.md`, создать/обновить `wiki/.source-hashes.json`, очистить прогресс, обновить `wiki/hot-context.md`, обновить `index.md` и `log.md`

### WIKI-INGEST-DOC: Ingestion документа / ADR

1. Прочитать файл по метке `docs` или `adr`
2. **Проверить `wiki/.ingest-progress.json`** — продолжить при наличии
3. Выделить: решение, контекст, рассмотренные альтернативы, итог
4. **Применить дерево решений** (слить или создать)
5. Создать/обновить `wiki/decisions/{slug}.md`
6. Добавить ссылки из связанных страниц modules/ и concepts/
7. **После страницы:** обновить `file-page-map.json` (док/ADR-файл → wiki/decisions/...), сохранить прогресс
8. **Выполнить каскадное обновление**
9. **В конце:** обновить `Индексировано` в `sources.md`, создать/обновить `wiki/.source-hashes.json`, очистить прогресс, обновить `wiki/hot-context.md`, обновить `index.md` и `log.md`

### WIKI-INGEST-PLAN: Ingestion плана разработки

1. Прочитать файл по метке `plans`
2. **Проверить `wiki/.ingest-progress.json`** — продолжить при наличии
3. Сопоставить задачи с существующими страницами вики
4. **Применить дерево решений** (слить или создать)
5. Создать/обновить `wiki/plans/{имя-плана}.md` с таблицей задач
6. Добавить ссылки из связанных модулей
7. **После страницы:** обновить `file-page-map.json` (план-файл → wiki/plans/...), сохранить прогресс
8. **В конце:** обновить `Индексировано` в `sources.md`, создать/обновить `wiki/.source-hashes.json`, очистить прогресс, обновить `wiki/hot-context.md`, обновить `index.md` и `log.md`

### WIKI-INGEST-TESTS: Ingestion тестов

1. Прочитать тесты по метке `src-test`
2. **Проверить `wiki/.ingest-progress.json`** — продолжить при наличии
3. Построить карту: что покрыто, что нет, критические пути без тестов
4. Создать/обновить `wiki/qa/coverage.md`
5. Пометить пробелы как tech-debt
6. **После страницы:** обновить `file-page-map.json` (тест-файл → wiki/qa/...), сохранить прогресс
7. **Выполнить каскадное обновление** (проверить затронутые modules/)
8. **В конце:** обновить `Индексировано` в `sources.md`, создать/обновить `wiki/.source-hashes.json`, очистить прогресс, обновить `wiki/hot-context.md`, обновить `index.md` и `log.md`

## Формат страницы вики

```markdown
# Заголовок

**Краткое описание**: Одно-два предложения.

**Источники**: относительные пути к исходным файлам (из sources.md).

**Статус**: active <!-- draft | active | stale | archived -->

**Уверенность**: extracted <!-- extracted | inferred | ambiguous -->

**Последнее обновление**: ГГГГ-ММ-ДД

---

Основной контент. Ссылки через [[вики-ссылки]].

## Противоречия
<!-- Заполняется только при наличии конфликтов между источниками -->
- **Вопрос X**: источник А утверждает [одно] (источник: path/a),
  источник Б утверждает [другое] (источник: path/b). Не разрешено.

## Tech debt
- [ ] Пункт → [[решение]] или [[план]]

## Связанные страницы
- [[страница-1]]
- [[страница-2]]
```

Секция `## Противоречия` добавляется только при наличии конфликтов.
Если конфликт охватывает две разные страницы — пометить в обеих с перекрёстными ссылками.

### Жизненный цикл страницы

Каждая страница вики проходит через состояния:

| Статус | Значение | Когда применяется |
|--------|----------|-------------------|
| `draft` | Черновик | Страница создана, но не проверена человеком |
| `active` | Актуальна | Страница проверена, источники в силе |
| `stale` | Устарела | Источники изменились, требуется ревью |
| `archived` | В архиве | Компонент удалён, решение отменено |

При `/wiki-lint`: страницы со статусом `stale` старше 30 дней — 🔴 критично; `draft` старше 7 дней — 🟡 внимание.

Переходы:
- `draft → active` — при первом /wiki-lint без ошибок
- `active → stale` — при `/wiki-update`, если глобальное изменение
- `stale → active` — после проверки человеком
- `→ archived` — только по решению человека

## Теги уверенности (`**Уверенность**`)

Каждая wiki-страница содержит поле `**Уверенность**`, отражающее надёжность информации:

| Тег | Значение | Когда применять |
|-----|----------|-----------------|
| `extracted` | Факт взят напрямую из кода/конфига | Класс существует, метод задокументирован, конфиг прочитан |
| `inferred` | Агент предположил из контекста | Назначение модуля, неявные связи между компонентами |
| `ambiguous` | Неясно, требует проверки человеком | Конфликт источников, устаревшая документация, нет тестов |

Правила:
- При `/wiki-ingest` — проставить `extracted` для прямых фактов из кода, `inferred` для умозаключений.
- При `/wiki-lint` — страницы с `inferred` + статус `draft` → 🟡 ВНИМАНИЕ (требует человеческой проверки).
- При `/wiki-ask` + архивировании ответа — использовать `inferred` (ответ основан на wiki, а не коде).

## Ответы на вопросы (WIKI-ASK)

1. Читать `wiki/index.md` (или `wiki/hot-context.md` если свежий, < 24ч) → найти релевантные страницы
2. Читать страницы; **не перечитывать исходный код** (если не помечено `[устарело]`)
3. Учитывать жизненный цикл страницы: stale → пометить устаревшей; draft → пометить непроверенной; archived → пометить архивной
4. Ответить со ссылками на конкретные страницы вики
5. Если ответа нет — сказать явно, предложить ingest нужного источника
6. После ответа предложить: **«Сохранить ответ в вики?»**

### Архивирование ответа

Если пользователь согласился сохранить ответ:

1. Определить категорию: в какой раздел вики лучше всего ложится ответ
2. **Применить дерево решений**: ответ дополняет существующую страницу → слить;
   отдельная самостоятельная тема → создать новую
3. При создании новой страницы:
   - Имя файла отражает тему вопроса: `wiki/{категория}/{тема-вопроса}.md`
   - В поле **Источники** указать страницы вики, на которых основан ответ
   - Добавить пометку `[Архив]` в заголовок краткого описания
   - Проставить **Уверенность**: `inferred` (ответ основан на wiki, а не на прямом чтении кода)
   - Проставить **Тип**: `q&a-archive` (страница создана из Q&A, не из /wiki-ingest)
4. Обновить `wiki/index.md`, добавить запись в `wiki/log.md`

Пометка **Тип**: `q&a-archive` сигнализирует агенту при следующем `/wiki-ask`:
«эта страница — архивный ответ, а не первичный источник; не доверять как `active` без перепроверки
исходных страниц, указанных в поле Источники».

## Repo-map: структурная карта кодовой базы

`wiki/architecture/repo-map.md` — автогенерируемая страница, которую создаёт скрипт,
а не Agents вручную. Она показывает низкоуровневый скелет: классы, интерфейсы, методы,
функции — то, что находится между высокоуровневыми wiki/modules/ и исходным кодом.

```
wiki/modules/payment-service.md   ← ЧТО делает: ответственность, архитектура
wiki/architecture/repo-map.md     ← КАК выглядит: классы, методы, сигнатуры  ← ЗДЕСЬ
wiki/architecture/file-page-map.json ← КУДА: маппинг исходный файл → wiki-страница
src/payment/PaymentService.java   ← реализация: полный код
```

### Система шестерён (adaptive detail)

Скрипт автоматически выбирает уровень детализации:

| Gear | Файлов | Показывает |
|------|--------|------------|
| 1 | < 30 | Все классы + полные сигнатуры методов с параметрами и возвращаемым типом |
| 2 | 30–150 | Все классы + имена публичных методов |
| 3 | > 150 | Структура папок + только точки входа |

**Гибридный режим**: для папок с ≥30 файлами принудительно включается Gear 1
(подробные сигнатуры), независимо от глобального gear. Папки помечаются `(подробно, N файлов)`.
Это даёт детальную карту крупных модулей внутри глобально большого проекта без потери контекста.

### Инкрементальный режим

Флаг `--incremental` ускоряет повторные запуски: скрипт сохраняет кэш
`wiki/architecture/.repo-map-cache.json` с mtime и parsed-типами каждого файла.
При следующем запуске перепарсиваются только файлы с изменившимся mtime.
Неизменённые файлы берутся из кэша. Если кэша нет — выполняется полный пересбор.

### Дополнительные артефакты

Скрипт также создаёт пустой шаблон `wiki/architecture/file-page-map.json` — словарь, который Agents заполняет во время `/wiki-ingest`:
```
{
  "src/main/java/com/example/PaymentService.java": "wiki/modules/payment-service.md",
  "src/main/java/com/example/OrderRepository.java": "wiki/modules/order-service.md"
}
```
Ключ — относительный путь исходного файла. Значение — wiki-страница модуля.
**Заполняется Agents во время `/wiki-ingest`** (не эвристикой). Используется `/wiki-update` для точечного lookup.

### Запуск

```bash
python wiki/scripts/repo-map.py                          # автоопределение из sources.md
python wiki/scripts/repo-map.py src/ --gear 2             # явный путь и шестерня
python wiki/scripts/repo-map.py src/ --incremental        # инкрементальный режим (по mtime)
bash wiki/scripts/repo-map.sh                             # bash-версия без зависимостей
```

### Правила работы с repo-map

- `repo-map.md` **не редактируется вручную** — только через скрипт
- Запускать при каждом `/wiki-update` **перед** анализом изменений (автоматически)
- Diff старого и нового repo-map = точный список структурных изменений
- Agents использует repo-map как **навигатор** при ingestion:
  читает скелет сначала, затем погружается только в нужные файлы
- Если repo-map показывает классы без wiki-страниц → предложить `/wiki-ingest`
- `file-page-map.json` обновляется только при `/wiki-ingest`, не при repo-map регенерации

## Прогресс индексации (wiki/.ingest-progress.json)

Файл чекпоинтов для возобновления прерванной индексации. Создаётся и обновляется автоматически при `/wiki-ingest`.

```json
{
  "started_at": "2026-06-10T10:30:00Z",
  "last_update": "2026-06-10T10:45:00Z",
  "current_label": "src-main",
  "phase": "modules",
  "processed_files": [
    "src/main/java/com/example/PaymentService.java",
    "src/main/java/com/example/OrderRepository.java"
  ],
  "total_files": 42,
  "current_file_index": 2
}
```

**Фазы:** `modules` → `concepts` → `architecture` → `index`

**Правила:**
- При старте `/wiki-ingest` — проверить файл. Если есть и не старше 24ч → предложить продолжить.
- После каждой обработанной wiki-страницы — добавить файл в `processed_files`, обновить `current_file_index`, `last_update`.
- При успешном завершении метки — **удалить файл** (прогресс не нужен).
- Если пользователь отказывается продолжать — удалить файл, начать чисто.

## Колонка `Индексировано` в sources.md

Автоматически обновляется при успешном завершении `/wiki-ingest` для метки.
Формат: `ГГГГ-ММ-ДД` (ISO дата).

Используется в `/wiki-update` как fallback дата для mtime-based detection когда git недоступен:

**Linux/macOS:**
```bash
find {путь из sources.md} -newer {дата_индексации} -type f
```

**Windows (PowerShell):**
```powershell
Get-ChildItem -Path {путь} -Recurse -File | Where-Object { $_.LastWriteTime -gt (Get-Date "{дата_индексации}") }
```

## Сценарий WIKI-UPDATE

Полный протокол обнаружения изменений, классификации и точечного обновления — в команде:

```
.config/opencode/commands/wiki-update.md
```

Краткая схема:
1. **Регенерация repo-map**: `python wiki/scripts/repo-map.py`
2. **Обнаружение изменений** через git diff (или SHA-256 / mtime fallback)
3. **Lookup через file-page-map.json** — точное попадание: файл → wiki-страница
4. **Классификация**: точечное / структурное / глобальное / удаление
5. **Lifecycle**: проставить **Статус** (active/stale) на затронутых страницах
6. **Сводка и подтверждение** перед записью
7. **Каскадное обновление** затронутых страниц
8. **Пересчёт SHA-256** затронутых файлов в `.source-hashes.json`
9. **Обновление hot-context.md** и запись в wiki/log.md

> ⚠️ **Важно**: `[устарело]` ставится ТОЛЬКО при глобальных изменениях (удаление компонента, слияние, переработка архитектуры). Точечные и структурные изменения обновляют секции напрямую, без инвалидации всей страницы.

## Правила цитирования и пометок

- Каждый факт: `(источник: путь/до/файла)`
- Два источника противоречат — зафиксировать в секции `## Противоречия`
  с обеими ссылками и пометкой «Не разрешено»
- Противоречие в двух разных страницах — пометить в обеих с перекрёстными ссылками
- Нет источника — пометить `[требует проверки]`

### Три уровня пометок при изменении кода

| Пометка | Когда ставить | Что означает |
|---|---|---|
| *(нет пометки)* | Точечное изменение — секция обновлена напрямую | Страница актуальна |
| `[изменено: filename]` | Структурное изменение, секция обновлена, но нужен взгляд человека | Проверить конкретный файл |
| `[устарело — проверить по источнику]` | Глобальное изменение: удалён компонент, слияние, переработка архитектуры | Требуется полный re-ingest |

`[устарело]` — не дефолтный ответ на любое изменение, а исключение для глобальных случаев.

## L1 hot cache (wiki/hot-context.md)

Краткая выжимка текущего состояния проекта — альтернатива полному сканированию index.md.

**Формат** — не больше 10 строк:

```markdown
# Hot context — {PROJECT_NAME}

**Последнее обновление**: ГГГГ-ММ-ДД ЧЧ:ММ

**Активные модули** (изменялись за последние 7 дней):
- [[modules/{module}]] — что изменилось
- [[modules/{module}]] — что изменилось

**Открытые проблемы**:
- [[страница]] :: описание

**Последнее действие**: {что сделано, кем}
```

**Правила работы:**
- Читать при `/wiki-start` **вместо** index.md (только если hot-context свежий, < 24ч)
- Если hot-context старше 24ч — читать полный index.md
- Обновлять при каждом `/wiki-update` и `/wiki-lint`
- Если изменился статус страницы (draft→active, active→stale) — отразить в hot-context

## SHA-256 верификация источников

Для точного детекта изменений (альтернатива mtime и git):

**wiki/.source-hashes.json** — словарь `относительный_путь → sha256`:

```json
{
  "src/main/java/com/example/PaymentService.java": "a1b2c3d4e5f6...",
  "src/main/java/com/example/OrderService.java": "f6e5d4c3b2a1..."
}
```

**Правила работы:**
- Создаётся при успешном `/wiki-ingest` метки (хеши всех прочитанных файлов)
- При `/wiki-update` — пересчитать хеши и сравнить со словарём
- Совпадает → файл не менялся
- Не совпадает → файл изменён (классифицировать: точечное/структурное/глобальное)
- Новый файл в маппинге → ДОБАВЛЕНИЕ
- SHA-256 считается командой (кроссплатформенно):
  - Linux/macOS: `sha256sum {файл}`
  - Windows: `Get-FileHash {файл} -Algorithm SHA256`
- Файл `.source-hashes.json` хранится рядом с `sources.md` в `wiki/`

## Аудит вики (WIKI-LINT)

Два уровня: авто-исправления (делать сразу) и доклад (не трогать, три яруса).

### Уровень 1 — Авто-исправления (делать сразу)

- Файл в wiki/ без записи в index.md → добавить запись
- Запись в index.md на несуществующий файл → пометить `[ОТСУТСТВУЕТ]`
- Битая [[ссылка]] с единственным совпадением → исправить путь
- Очевидно связанные страницы без взаимных ссылок → добавить
- Страница со статусом **draft** без ошибок → продвинуть до **active**
- Страница со статусом **stale**, чьи источники не менялись → вернуть **active**

### Уровень 2 — Доклад (не трогать, три яруса)

#### 🔴 Критично — требует решения команды
- Открытое противоречие в `## Противоречия` старше 30 дней
- Страница `[устарело]` или статус **stale** старше 30 дней без обновления
- Метка из sources.md без ни одной wiki-страницы

#### 🟡 Внимание — исправить в ближайший спринт
- Страница помечена `[изменено:]` или `[требует проверки]`
- Страница со статусом **draft** старше 7 дней
- Осиротевшая страница (нет входящих ссылок)
- Компонент упоминается на странице, но нет своей страницы modules/
- `overview.md` не обновлялась > 30 дней

#### 🔵 Информация — улучшить по возможности
- Страница без блока `## Связанные страницы`
- Страница без поля `Источники`
- Страница без поля **Статус** (не проставлен lifecycle)
- Архивные страницы, чьи источники обновились

Вывод lint-отчёта двумя блоками:
```
ИСПРАВЛЕНО АВТОМАТИЧЕСКИ (N): ...

ТРЕБУЕТ ВНИМАНИЯ:
  🔴 Критично (N): ...
  🟡 Внимание (N): ...
  🔵 Информация (N): ...

Итог: N авто-исправлений | N🔴 N🟡 N🔵
```

## Правила

- **Wiki first**: перед любой задачей — читать wiki. Не код.
- **Code → Wiki**: нашёл в коде что-то отсутствующее в wiki → записать немедленно.
- **Never from memory**: Agents не отвечает о проекте из собственных знаний — только из wiki.
- Никогда не изменять файлы из `wiki/sources.md`
- Всегда обновлять `wiki/index.md` и `wiki/log.md` после любых изменений
- Имена файлов: строчные буквы с дефисами (`order-service.md`)
- Не вставлять большие блоки кода — ссылаться на исходный файл
- Перед созданием страницы применять дерево решений (слить vs создать)
- После каждого ingestion выполнять каскадное обновление
- Противоречия фиксировать в секции `## Противоречия` с обеими ссылками
- При неопределённости с категоризацией — спрашивать пользователя
