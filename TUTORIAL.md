# Wiki-скилл для проектов: руководство разработчика

---

## 1. Для чего нужен скилл

Каждый раз, открывая новую сессию с LLM, разработчик вынужден снова объяснять контекст: какие компоненты существуют, как устроена архитектура, какие решения уже приняты. Это утомляет и отнимает время. Традиционный выход — RAG-индексирование репозитория — требует векторной базы данных, конвейера обновлений и отдельной инфраструктуры.

Скилл предлагает другой подход. Вместо поиска по векторам Agents читает заранее подготовленные страницы на человеческом языке — краткие, структурированные, взаимосвязанные. Накопленные знания о проекте живут прямо в репозитории рядом с исходниками. Открыл сессию, написал `/wiki-start` — и ассистент уже знает всё необходимое.

Скилл решает три задачи. Первая: устранить потерю контекста между сессиями. Вторая: дать команде единое место, где зафиксированы решения, зависимости и технический долг. Третья: сделать работу с LLM в долгосрочном проекте управляемой и предсказуемой.

Подход не зависит от языка программирования или типа проекта. Кодовая база на Python, Go, Java, TypeScript, Rust — для вики это всё одинаковые источники, которые нужно прочитать, понять и превратить в страницы.

## 2. Как установить

### 2.1 Разместить глобальный SKILL.md

Скилл хранится в одном месте и подключается к любому проекту по ссылке.

```
Linux / macOS:
  ~/.opencode/skills/wiki/SKILL.md

Windows:
  C:\Users\{username}\.config\opencode\skills\wiki\SKILL.md
```

Скопируйте файл `SKILL.md` по этому пути. Обновлять его придётся только при изменении самого процесса — не при работе с каждым отдельным проектом.

### 2.2 Добавить файлы в проект

Структура, которую нужно создать в корне репозитория:

```
{корень проекта}/
  AGENTS.md                    ← инструкция для Agents (15–20 строк)

  wiki/scripts/
    repo-map.py                ← структурная карта + file-page-map.json
    repo-map.sh                ← bash-версия без зависимостей
    install-hook.sh            ← установщик git post-commit хука

  wiki/
    sources.md                 ← манифест: что считается источниками
    index.md                   ← оглавление вики (поначалу пустое)
    log.md                     ← журнал изменений (поначалу пустой)
    hot-context.md             ← L1 кэш: краткая выжимка (< 24ч)
    .source-hashes.json        ← SHA-256 хеши источников (создаётся при ingestion)
    .ingest-progress.json      ← чекпоинты для возобновления прерванного ingestion
    architecture/
      repo-map.md              ← автогенерируемый скелет классов/методов
      file-page-map.json       ← маппинг: исходный файл → wiki-страница

  opencode/
    commands/                  ← 8 slash-команд
      wiki-start.md
      wiki-ingest.md
      wiki-ask.md
      wiki-update.md
      wiki-repomap.md
      wiki-lint.md
      wiki-stats.md
      wiki-plan.md
```

Исходная структура репозитория остаётся нетронутой. Новые файлы добавляются рядом, не конфликтуя ни с одним инструментом сборки или деплоя.

### 2.3 Заполнить AGENTS.md

Минимальный `AGENTS.md` содержит три блока: метаданные проекта, путь к скиллу и указание на манифест источников.

```markdown
# AGENTS.md — {PROJECT_NAME}

## Проект
- **Название**: {PROJECT_NAME}
- **Тип**: {кодовая база / конфигурационный / гибридный}
- **Стек / технологии**: {перечислить кратко}
- **Описание**: {одно предложение о назначении}

## Wiki-навык
Для всех операций с вики читай: ~/.opencode/skills/wiki/SKILL.md

## Источники
Манифест исходных данных: wiki/sources.md

## Старт сессии
1. Прочитай wiki/sources.md
2. Прочитай wiki/hot-context.md (если свежий, < 24ч) или wiki/index.md (если устарел)
3. Кратко сообщи: что знаешь, что может быть устаревшим
```

### 2.4 Заполнить sources.md

Откройте `wiki/sources.md` и пропишите реальные пути своего проекта. Каждая строка таблицы — метка, по которой Agents находит нужную директорию или файл.

Колонка **Индексировано** — заполняется автоматически при успешном `/wiki-ingest`.

**Пример для проекта с исходным кодом:**

```markdown
# Манифест источников

## Исходный код
| Метка      | Путь      | Описание              | Индексировано |
|------------|-----------|-----------------------|---------------|
| `src-main` | `src/`    | Основной код          | 2026-06-10    |
| `src-test` | `tests/`  | Тесты                 | —             |

## Конфигурации
| Метка        | Путь        | Описание              | Индексировано |
|--------------|-------------|-----------------------|---------------|
| `config-app` | `config/`   | Конфигурация          | —             |

## Документация
| Метка  | Путь        | Описание                 | Индексировано |
|--------|-------------|--------------------------|---------------|
| `docs` | `docs/`     | Техническая документация | —             |
| `adr`  | `docs/adr/` | Архитектурные решения    | —             |

## Исключения
- .git/ .idea/ __pycache__/ target/ build/ dist/ *.pyc *.class *.jar wiki/
```

**Пример для конфигурационного проекта (XML / YAML / DSL):**

```markdown
# Манифест источников

## Конфигурации сервисов
| Метка          | Путь               | Описание                        | Индексировано |
|----------------|--------------------|---------------------------------|---------------|
| `cfg-routes`   | `config/routes/`   | Маршруты и правила обработки    | —             |
| `cfg-services` | `config/services/` | Описания развёртываемых сервисов| —             |

## Исключения
- .git/ *.bak *.tmp wiki/
```

### 2.5 Установить git-хук

Один раз в корне проекта:

```bash
sh wiki/scripts/install-hook.sh
```

После установки `repo-map.md` обновляется автоматически в фоне после каждого `git commit`. Хук не переносится при клонировании — каждый разработчик запускает `install-hook.sh` у себя.

### 2.6 Установить tree-sitter (опционально)

`repo-map.py` поддерживает два режима парсинга:

| Режим | Как активируется | Языки |
|---|---|---|
| **Tree-sitter** (приоритет) | pip install + автоопределение | TypeScript, TSX, JS, Rust, C#, Swift, Ruby, C, C++, Scala, PHP, Dart, Lua + Python, Java, Go, Kotlin |
| **Regex** (fallback) | всегда, без зависимостей | Python, Java, Go, Kotlin, XML |

Если tree-sitter не установлен — скрипт тихо переключается на regex. Поведение для Java/Python/Go не меняется.

Установка пакетов под свой стек:

```bash
# TypeScript / JavaScript проект
pip install tree-sitter tree-sitter-typescript tree-sitter-javascript

# Rust проект
pip install tree-sitter tree-sitter-rust

# C# / .NET проект
pip install tree-sitter tree-sitter-c-sharp

# Всё сразу (полный набор)
pip install tree-sitter \
    tree-sitter-typescript tree-sitter-javascript \
    tree-sitter-rust tree-sitter-c-sharp \
    tree-sitter-go tree-sitter-java tree-sitter-python \
    tree-sitter-kotlin tree-sitter-ruby tree-sitter-swift \
    tree-sitter-c tree-sitter-cpp tree-sitter-scala \
    tree-sitter-php tree-sitter-dart
```

Принудительно отключить tree-sitter (только regex):

```bash
python wiki/scripts/repo-map.py --no-ast
```

### 2.7 Добавить в .gitignore (опционально)

Вики полезна всей команде — её стоит хранить в репозитории. Если отдельные страницы содержат чувствительные сведения, исключите их точечно:

```
wiki/env/production.md
wiki/env/secrets.md
```

## 3. Как использовать

Работа со скиллом строится вокруг восьми slash-команд.

| Команда | Назначение |
|---|---|
| `/wiki-start [задача]` | Восстановить контекст (L1 hot cache). С аргументом — сразу в задачу через wiki-first |
| `/wiki-ingest <метка>` | Проиндексировать источник с созданием SHA-256 хешей и проставлением lifecycle |
| `/wiki-ask <вопрос>` | Задать вопрос о проекте через вики; при необходимости сохранить ответ |
| `/wiki-update` | Авто-обнаружение изменений через git/SHA-256 + точечный lookup через `file-page-map.json` |
| `/wiki-repomap` | Регенерация структурного скелета (`repo-map.md` + `file-page-map.json` шаблон) |
| `/wiki-lint` | Аудит: 🔴 критично / 🟡 внимание / 🔵 информация + lifecycle + граф вики |
| `/wiki-stats` | Сводка: страниц, lifecycle, покрытие, осиротевшие, инфраструктура |
| `/wiki-plan <план: задача → статус>` | Обновить прогресс по плану |

## 4. Ключевые механизмы скилла

### Дерево решений: слить или создать

При каждом ingestion Agents не создаёт новую страницу автоматически. Сначала проверяется: существует ли уже страница, покрывающая ту же суть?

```
Источник уже проиндексирован?
  ДА → страница покрывает ту же суть?
         ДА  → слить данные в существующую (обновить Sources и разделы)
         НЕТ → создать новую + добавить перекрёстные ссылки
  НЕТ → создать новую
```

### Каскадное обновление

После любого изменения страницы Agents сканирует связанные разделы. Добавили новый метод в модуль — обновляется `architecture/module-map.md`. Изменилась зависимость — обновляется `concepts/` связанного понятия.

### Протокол противоречий

Если два источника утверждают разное — конфликт фиксируется явно:

```markdown
## Противоречия
- **Поведение при ошибке**: config/routes/billing.xml задаёт retry=3,
  docs/adr/adr-005.md предписывает fail-fast без повторов.
  Не разрешено — требует решения команды.
```

### Двухуровневый аудит

`/wiki-lint` делит проверки на три уровня. Добавлены структурные проверки графа:

| Уровень | Метка | Примеры |
|---------|-------|---------|
| 🔴 Критично | `[КРИТИЧНО]` | Открытое противоречие > 30d, stale > 30d, метка без страниц |
| 🟡 Внимание | `[ВНИМАНИЕ]` | Осиротевшая страница, draft > 7d, `inferred + draft`, **hub-stubs**, **orphan clusters** |
| 🔵 Информация | `[ИНФО]` | Нет **Связанные страницы**, нет **Источники**, нет **Статус**, нет **Уверенность** |

**Hub-stub** — страница с >5 входящих [[wikilinks]] и <300 символов контента. Часто: создаётся узловая `[[auth]]` без контента, и все на неё ссылаются.

**Orphan cluster** — группа страниц, связанных между собой, но без связей с основным графом вики.

### Теги уверенности

Каждая wiki-страница содержит поле `**Уверенность**`:

| Тег | Значение |
|-----|----------|
| `extracted` | Факт взят напрямую из кода/конфига |
| `inferred` | Агент предположил из контекста |
| `ambiguous` | Неясно, требует проверки человеком |

Страницы с `inferred + draft` выделяются в `/wiki-lint` как 🟡 ВНИМАНИЕ. Страницы из `/wiki-ask` получают `inferred` автоматически.

### Q&A-archive

Ответы, сохранённые через `/wiki-ask`, получают пометку `**Тип**: q&a-archive`. При следующем `/wiki-ask` агент знает, что это не первичный источник из кода — и перепроверяет по Источникам страницы, если нужна точность.

### L1 Hot Cache

`wiki/hot-context.md` — краткая выжимка контекста (до ~10 строк с ключевыми ссылками). Читается при `/wiki-start` вместо полного `index.md`. Если hot-context свежий (< 24ч) — старт сессии занимает секунды.

### Page Lifecycle

Каждая страница вики имеет поле **Статус**:

```
draft → active → stale → archived
```

- **draft** — новая страница, требует проверки. Если draft > 7d — `/wiki-lint` покажет 🟡.
- **active** — актуальная страница.
- **stale** — требует обновления. Если stale > 30d — 🔴.
- **archived** — не участвует в `/wiki-ask`.

### SHA-256 хеширование

`wiki/.source-hashes.json` — словарь `{ "путь/к/файлу": "sha256hex" }`. Приоритетный метод в `/wiki-update`, работает без git и одинаково на всех платформах.

### 4.1 Repo-map: структурная карта кодовой базы

Repo-map — автогенерируемый скелет всего проекта. Скрипт проходит по файлам, извлекает классы, методы и функции, адаптирует детализацию под размер кодовой базы.

```
wiki/modules/payment-service.md      ← ЧТО делает: ответственность, архитектура
wiki/architecture/repo-map.md        ← КАК выглядит: классы, методы, сигнатуры
wiki/architecture/file-page-map.json ← КУДА: маппинг исходный файл → wiki-страница
src/payment/PaymentService.java      ← реализация: полный код
```

**Режимы парсинга** (в порядке приоритета):

1. **Tree-sitter** — если установлены пакеты (31+ язык, AST-точность)
2. **Regex-fallback** — всегда работает, без зависимостей (Python, Java, Go, Kotlin, XML)

Детализация (система шестерён):

| Gear | Файлов | Что показывает |
|------|--------|----------------|
| 1 | < 30 | Все классы + полные сигнатуры методов |
| 2 | 30–150 | Все классы + имена публичных методов |
| 3 | > 150 | Структура папок + точки входа |

Запуск:

```bash
python wiki/scripts/repo-map.py           # tree-sitter (если установлен) + regex-fallback
python wiki/scripts/repo-map.py --no-ast  # принудительно только regex
python wiki/scripts/repo-map.py --gear 1  # принудительная шестерня
python wiki/scripts/repo-map.py --incremental  # только изменённые файлы
```

Или через slash-команду:

```
/wiki-repomap
```

### 4.2 Git post-commit хук

После установки хука `repo-map.md` обновляется автоматически в фоне после каждого коммита:

```bash
sh wiki/scripts/install-hook.sh   # один раз, в корне проекта
```

Хук запускает только `repo-map.py --incremental` — без LLM, завершается за секунды. Не блокирует `git commit`. `/wiki-update` по-прежнему запускается вручную.

## 5. Шаги использования

### Шаг 1 — Первичная индексация (выполняется один раз)

```bash
python wiki/scripts/repo-map.py    # repo-map.md + file-page-map.json (пустой шаблон)
```

Затем — ingestion. Начинайте с основного кода:

```
/wiki-ingest src-main
/wiki-ingest src-test
/wiki-ingest config-app
/wiki-ingest adr
/wiki-ingest plans
```

После завершения:

```
/wiki-lint
```

При большом количестве файлов ingestion может прерваться — создаётся `wiki/.ingest-progress.json`. При следующем запуске Agents предложит продолжить с прерванного места.

### Шаг 2 — Начало каждой рабочей сессии

```
/wiki-start
/wiki-start добавить рефанд в платёжный модуль   ← сразу в задачу
```

### Шаг 3 — Ежедневная работа

```
/wiki-ask как настроена обработка ошибок в платёжном модуле?
→ [Agents отвечает с цитатами]
→ «Сохранить ответ в вики» — Agents архивирует как страницу с Тип: q&a-archive

/wiki-update          # авто-детекция: git diff или SHA-256
/wiki-repomap         # если добавились/удалились классы
/wiki-plan sprint-5: задача №3 → в работе
```

### Шаг 4 — Поддержка актуальности

Правило простое: изменил код — сразу `/wiki-update`. Добавил/удалил классы — `/wiki-repomap`. Раз в спринт — `/wiki-lint` (двухуровневый аудит: авто-исправления + доклад 🔴🟡🔵 + граф вики).

## 6. Когда в проекте нужно вызывать скилл

**Начало сессии.** Любой новый разговор о проекте стартует с `/wiki-start`.

**После изменения источников.** Поправили конфигурацию, переписали модуль — `/wiki-update`. Большинство правок обновляют страницу напрямую без пометки `[устарело]`.

**При появлении нового компонента.** Добавили модуль — занесите путь в `sources.md`, затем `/wiki-ingest` с новой меткой.

**При обнаружении противоречия.** Нашли конфликт — немедленно зафиксируйте через `/wiki-update`. Секция `## Противоречия` останется до решения команды.

**При принятии архитектурного решения.** Зафиксировали в ADR — сразу `/wiki-ingest adr`.

**При онбординге.** Три файла: `AGENTS.md`, `wiki/sources.md`, `wiki/index.md`. Новый участник запускает `/wiki-start`.

**Для оценки состояния вики.** `/wiki-stats` покажет сводку. `/wiki-lint` — граф + lifecycle + hub-stubs + orphan clusters.

**После структурных изменений в коде.** `/wiki-repomap` — diff покажет, что именно изменилось. Если хук установлен — обновление происходит автоматически после коммита.

## 7. Последовательность запуска команд

### 7.1 Первичная настройка нового проекта

```
1. Создать файлы:
   AGENTS.md
   wiki/sources.md          ← заполнить пути под свой проект
   wiki/index.md            ← пустой
   wiki/log.md              ← пустой
   wiki/hot-context.md      ← пустой
   wiki/scripts/repo-map.py      ← скопировать
   wiki/scripts/install-hook.sh  ← скопировать
   opencode/commands/            ← скопировать все 8 команд

2. sh wiki/scripts/install-hook.sh   ← установить git-хук

3. python wiki/scripts/repo-map.py   ← создаёт repo-map.md + file-page-map.json

4. /wiki-ingest <первичный источник>
   → Обсудить с Agents: находки, противоречия, терминология
   → Подтвердить создание страниц (Статус: draft)
   → file-page-map.json заполняется

5. /wiki-ingest <следующий источник>
   → [каскадное обновление выполняется автоматически]

6. Повторить для всех меток из sources.md

7. /wiki-lint
   → 🔴🟡🔵 аудит с lifecycle проверками + граф
   → Авто-исправления применятся сразу

8. Закоммитить wiki/ в репозиторий
```

### 7.2 Типичная рабочая сессия

```
Открыть сессию
  └─ /wiki-start
       └─ [Agents: контекст, устаревшее, открытые противоречия]

Работа с проектом
  ├─ /wiki-ask <вопрос>
  │    └─ [Agents отвечает] → «Сохранить?» → архивируется с Тип: q&a-archive
  └─ Внёс изменения в код
       └─ git commit                   ← хук обновляет repo-map автоматически
       └─ /wiki-update                 ← LLM-обновление wiki-страниц

Завершить сессию
  └─ (вики актуальна — следующая стартует с чистым контекстом)
```

### 7.3 Конец спринта

```
/wiki-stats
  └─ [Сводка: страниц по lifecycle, покрытие, осиротевшие]

/wiki-lint
  └─ [🔴🟡🔵 аудит применён]
  └─ [Hub-stubs и orphan clusters — рассмотреть с командой]

/wiki-plan <имя-плана>
  └─ Обновить статусы задач

/wiki-ingest <новые метки, если появились>
```

### 7.4 Онбординг нового разработчика

