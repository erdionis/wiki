# Wiki-скилл для проектов: руководство разработчика

---

## 1. Для чего нужен скилл

Каждый раз, открывая новую сессию с LLM, разработчик вынужден снова объяснять контекст: какие компоненты существуют, как устроена архитектура, какие решения уже приняты. Это утомляет и отнимает время. Традиционный выход — RAG-индексирование репозитория — требует векторной базы данных, конвейера обновлений и отдельной инфраструктуры.

Скилл предлагает другой подход. Вместо поиска по векторам Agents читает заранее подготовленные страницы на человеческом языке — краткие, структурированные, взаимосвязанные. Накопленные знания о проекте живут прямо в репозитории рядом с исходниками. Открыл сессию, написал `/wiki-start` — и ассистент уже знает всё необходимое.

Скилл решает три задачи. Первая: устранить потерю контекста между сессиями. Вторая: дать команде единое место, где зафиксированы решения, зависимости и технический долг. Третья: сделать работу с LLM в долгосрочном проекте управляемой и предсказуемой.

Подход не зависит от языка программирования или типа проекта. Кодовая база на Python, Go, Java, конфигурационные XML-файлы, разворачивающие готовый сервис, — для вики это всё одинаковые источники, которые нужно прочитать, понять и превратить в страницы.

## 2. Как установить

### 2.1 Разместить глобальный SKILL.md

Скилл хранится в одном месте и подключается к любому проекту по ссылке. Папку можно назвать произвольно — важно запомнить путь.

```
Linux / macOS:
  ~/.opencode/skills/wiki/SKILL.md

Windows:
  C:\Users\{username}\.config\opencode\skills\wiki\SKILL.md
```

Скопируйте файл `SKILL.md` по этому пути. Обновлять его придётся только при изменении самого процесса — не при работе с каждым отдельным проектом.

### 2.2 Добавить файлы в проект

Структура, которую нужно создать в корне репозитория:

```
{корень проекта}/
  AGENTS.md                    ← инструкция для Agents (15–20 строк)

  scripts/
    repo-map.py                ← структурная карта + file-page-map.json
    repo-map.sh                ← bash-версия без зависимостей

  wiki/
    sources.md                 ← манифест: что считается источниками
    index.md                   ← оглавление вики (поначалу пустое)
    log.md                     ← журнал изменений (поначалу пустой)
    hot-context.md             ← L1 кэш: краткая выжимка (< 24ч)
    .source-hashes.json        ← SHA-256 хеши источников (создаётся при ingestion)
    .ingest-progress.json      ← чекпоинты для возобновления прерванного ingestion
    architecture/
      repo-map.md              ← автогенерируемый скелет классов/методов
      file-page-map.json       ← маппинг: исходный файл → wiki-страница

  .opencode/
    commands/                  ← 8 slash-команд
      wiki-start.md
      wiki-ingest.md
      wiki-ask.md
      wiki-update.md
      wiki-repomap.md
      wiki-lint.md
      wiki-stats.md
      wiki-plan.md
```

Исходная структура репозитория остаётся нетронутой. Новые файлы добавляются рядом, не конфликтуя ни с одним инструментом сборки или деплоя.

### 2.3 Заполнить Agents.md

Минимальный `Agents.md` содержит три блока: метаданные проекта, путь к скиллу и указание на манифест источников.

```markdown
# Agents.md — {PROJECT_NAME}

## Проект
- **Название**: {PROJECT_NAME}
- **Тип**: {кодовая база / конфигурационный / гибридный}
- **Стек / технологии**: {перечислить кратко}
- **Описание**: {одно предложение о назначении}

## Wiki-навык
Для всех операций с вики читай: ~/.Agents/skills/wiki/SKILL.md

## Источники
Манифест исходных данных: wiki/sources.md

## Старт сессии
1. Прочитай wiki/sources.md
2. Прочитай wiki/hot-context.md (если свежий, < 24ч) или wiki/index.md (если устарел)
3. Кратко сообщи: что знаешь, что может быть устаревшим
```

### 2.4 Заполнить sources.md

Откройте `wiki/sources.md` и пропишите реальные пути своего проекта. Каждая строка таблицы — метка, по которой Agents находит нужную директорию или файл. Менять манифест будет только разработчик — ассистент его никогда не редактирует.

Колонка **Индексировано** — заполняется автоматически при успешном `/wiki-ingest` (дата последней индексации метки). Используется как fallback для `/wiki-update` когда git недоступен.

**Пример для проекта с исходным кодом:**

```markdown
# Манифест источников

## Исходный код
| Метка      | Путь      | Описание              | Индексировано |
|------------|-----------|-----------------------|---------------|
| `src-main` | `src/`    | Основной код          | 2026-06-10    |
| `src-test` | `tests/`  | Тесты                 | —             |

## Конфигурации
| Метка        | Путь        | Описание              | Индексировано |
|--------------|-------------|-----------------------|---------------|
| `config-app` | `config/`   | Конфигурация          | —             |
| `config-env` | `deploy/`   | Конфиги окружений     | —             |

## Документация
| Метка  | Путь        | Описание              | Индексировано |
|--------|-------------|-----------------------|---------------|
| `docs` | `docs/`     | Техническая документация | —          |
| `adr`  | `docs/adr/` | Архитектурные решения | —             |

## Планы
| Метка   | Путь      | Описание                    | Индексировано |
|---------|-----------|-----------------------------|---------------|
| `plans` | `plans/`  | Планы разработки, LLM-задачи| —             |

## Исключения
- .git/ .idea/ __pycache__/ target/ build/ dist/ *.pyc *.class *.jar wiki/
```

**Пример для конфигурационного проекта (XML / YAML / DSL):**

```markdown
# Манифест источников

## Конфигурации сервисов
| Метка          | Путь               | Описание                        | Индексировано |
|----------------|--------------------|---------------------------------|---------------|
| `cfg-routes`   | `config/routes/`   | Маршруты и правила обработки    | —             |
| `cfg-services` | `config/services/` | Описания развёртываемых сервисов| —             |
| `cfg-env-dev`  | `envs/dev/`        | Среда разработки                | —             |
| `cfg-env-prod` | `envs/prod/`       | Продакшн-среда                  | —             |

## Схемы
| Метка     | Путь        | Описание              | Индексировано |
|-----------|-------------|-----------------------|---------------|
| `schemas` | `schemas/`  | XSD / JSON Schema     | —             |

## Исключения
- .git/ *.bak *.tmp wiki/
```

### 2.5 Добавить в .gitignore (опционально)

Вики полезна всей команде — её стоит хранить в репозитории. Но если отдельные страницы содержат чувствительные сведения, исключите их точечно:

```
wiki/env/production.md
wiki/env/secrets.md
```

## 3. Как использовать

Работа со скиллом строится вокруг восьми slash-команд. Каждая решает конкретную задачу — не нужно помнить длинные промты или объяснять порядок действий заново в каждой сессии.

| Команда | Назначение |
|---|---|
| `/wiki-start [задача]` | Восстановить контекст (L1 hot cache). С аргументом — сразу в задачу через wiki-first |
| `/wiki-ingest <метка>` | Проиндексировать источник с созданием SHA-256 хешей и проставлением lifecycle |
| `/wiki-ask <вопрос>` | Задать вопрос о проекте через вики; при необходимости сохранить ответ |
| `/wiki-update` | Авто-обнаружение изменений через git/SHA-256 + точечный lookup через `file-page-map.json` + обновление lifecycle |
| `/wiki-repomap` | Регенерация структурного скелета (`repo-map.md` + `file-page-map.json` шаблон) |
| `/wiki-lint` | Аудит: 🔴 критично / 🟡 внимание / 🔵 информация + lifecycle проверки + hot-context |
| `/wiki-stats` | Сводка: страниц, lifecycle, покрытие, осиротевшие, инфраструктура |
| `/wiki-plan <план: задача → статус>` | Обновить прогресс по плану |

Вся команда пользуется одним набором инструкций. Новый участник, подключившись к проекту, сразу получает доступ к накопленным знаниям через те же команды.

## 4. Ключевые механизмы скилла

Прежде чем переходить к пошаговым инструкциям, важно понять внутренние механизмы, которые делают вики самосогласованной.

### Дерево решений: слить или создать

При каждом ingestion Agents не создаёт новую страницу автоматически. Сначала проверяется: существует ли уже страница, покрывающая ту же суть?

```
Источник уже проиндексирован?
  ДА → страница покрывает ту же суть?
         ДА  → слить данные в существующую (обновить Sources и разделы)
         НЕТ → создать новую + добавить перекрёстные ссылки
  НЕТ → создать новую
```

Это предотвращает дублирование и держит базу компактной.

### Каскадное обновление

После любого изменения страницы Agents сканирует связанные разделы на побочные эффекты. Добавили новый метод в модуль — автоматически обновляется `architecture/module-map.md`. Изменилась зависимость — обновляется `concepts/` связанного понятия. Все затронутые страницы получают свежую дату и попадают в `log.md`.

### Протокол противоречий

Если два источника утверждают разное — Agents не выбирает между ними самостоятельно. Конфликт фиксируется явно в секции `## Противоречия` на затронутой странице:

```markdown
## Противоречия
- **Поведение при ошибке**: config/routes/billing.xml задаёт retry=3
  (источник: config/routes/billing.xml), docs/adr/adr-005.md
  предписывает fail-fast без повторов (источник: docs/adr/adr-005.md).
  Не разрешено — требует решения команды.
```

Если конфликт касается двух разных страниц — пометка появляется в обеих с перекрёстными ссылками.

### Двухуровневый аудит

Команда `/wiki-lint` делит проверки на три уровня:

| Уровень | Метка | Примеры | Действие |
|---------|-------|---------|----------|
| 🔴 Критично | `[КРИТИЧНО]` | Открытое противоречие > 30d, stale > 30d, метка без страниц, страница [устарело] > 30d | Доклад, не исправлять |
| 🟡 Внимание | `[ВНИМАНИЕ]` | Осиротевшая страница, draft > 7d, [изменено:], overview > 30d | Доклад, не исправлять |
| 🔵 Информация | `[ИНФО]` | Отсутствует **Связанные страницы**, отсутствует **Источники**, нет **Статус** | Доклад + авто-исправление |

Первый уровень — авто-исправления: битые ссылки, отсутствующие записи в index, продвижение draft→active, возврат stale→active.
Второй уровень — доклад по трём ярусам 🔴🟡🔵. Agents исправляет только детерминированные проблемы первого уровня.
Эвристические — только докладывает, не трогает.

### L1 Hot Cache

`wiki/hot-context.md` — краткая выжимка контекста (до ~10 строк с ключевыми ссылками). Читается при `/wiki-start` вместо полного `index.md`. Если hot-context свежий (< 24ч по дате) — старт сессии занимает секунды. Если устарел — используется полный `index.md`, а затем hot-context обновляется.

### Page Lifecycle

Каждая страница вики имеет поле **Статус** с одним из четырёх состояний:

```
draft → active → stale → archived
```

- **draft** — новая страница, требует проверки человеком. Если draft > 7d — `/wiki-lint` покажет 🟡.
- **active** — актуальная страница. Переходит в stale при `/wiki-update`, если затронут её компонент.
- **stale** — требует обновления. Если stale > 30d — `/wiki-lint` покажет 🔴.
- **archived** — страница больше не актуальна, сохранена для истории. Не участвует в `/wiki-ask`.

Lifecycle позволяет точно определять здоровье вики и не полагаться на эвристики.

### SHA-256 хеширование

`wiki/.source-hashes.json` — словарь вида `{ "путь/к/файлу": "sha256hex" }`. Создаётся при `/wiki-ingest` и обновляется при `/wiki-update`. Используется как fallback, когда git недоступен — точнее, чем mtime, и работает одинаково на Windows/macOS/Linux.

---


### 4.1 Repo-map: структурная карта кодовой базы

RepoMap — автогенерируемый низкоуровневый скелет всего проекта. Скрипт проходит по файлам, извлекает классы, методы и функции, адаптирует детализацию под размер кодовой базы. Agents использует эту карту как навигатор при ingestion — вместо слепого чтения каждого файла.

```
wiki/modules/payment-service.md      ← ЧТО делает: ответственность, архитектура
wiki/architecture/repo-map.md        ← КАК выглядит: классы, методы, сигнатуры
wiki/architecture/file-page-map.json ← КУДА: маппинг исходный файл → wiki-страница
src/payment/PaymentService.java      ← реализация: полный код
```

Детализация автоматически подстраивается под размер проекта по системе шестерён:

| Gear | Файлов | Что показывает |
|------|--------|----------------|
| 1 | < 30 | Все классы + полные сигнатуры методов |
| 2 | 30–150 | Все классы + имена публичных методов |
| 3 | > 150 | Структура папок + точки входа |

Скрипт также создаёт пустой шаблон `wiki/architecture/file-page-map.json`:
```json
{}
```
**Заполняется Agents во время `/wiki-ingest`** (не эвристикой): для каждого проиндексированного файла Agents добавляет запись `"отн.путь/файла": "wiki/modules/имя-страницы.md"`.
Используется `/wiki-update` для точечного lookup — не нужно перечитывать все страницы, достаточно проверить только затронутые.

Генерируется скриптом — не Agents:

```bash
python wiki/scripts/repo-map.py          # Python-версия (рекомендуется)
bash wiki/scripts/repo-map.sh            # bash-версия без зависимостей
python wiki/scripts/repo-map.py --gear 1 # принудительная шестерня
```

Или через slash-команду:

```
/wiki-repomap
```

Страница `wiki/architecture/repo-map.md` создаётся скриптом и **не редактируется вручную**.
`file-page-map.json` обновляется только при `/wiki-ingest`, не при регенерации repo-map.
Обновляйте repo-map каждый раз при `/wiki-update` — diff старого и нового точно показывает,
что структурно изменилось в проекте.

## 5. Шаги использования

### Шаг 1 — Первичная индексация (выполняется один раз)

Перед тем как вики заработает, нужно создать структурную карту и наполнить вики содержимым.

**Сначала — repo-map:**
```bash
python wiki/scripts/repo-map.py    # repo-map.md + file-page-map.json (пустой шаблон)
```
Или через slash-команду:
```
/wiki-repomap
```

**Затем — ingestion.** Agents читает исходные файлы, создаёт страницы с lifecycle-статусом **draft**, заполняет `file-page-map.json` и `.source-hashes.json`.
Начинайте с основного кода или главных конфигов, затем переходите к компонентам поочерёдно. Торопиться не стоит: после каждого ingestion ассистент применяет каскадное обновление и предлагает обсудить находки — это момент зафиксировать нюансы, невидимые напрямую из файлов.

**Важно:** при большом количестве файлов ingestion может прерваться. В этом случае создаётся файл `wiki/.ingest-progress.json` с чекпоинтом. При следующем запуске `/wiki-ingest` той же метки Agents предложит продолжить с прерванного места — не нужно начинать заново.

```
/wiki-ingest src-main      # или cfg-routes для конфигурационного проекта
/wiki-ingest src-test
/wiki-ingest config-app
/wiki-ingest config-env
/wiki-ingest adr           # если есть
/wiki-ingest plans         # если есть
```

После завершения — аудит:

```
/wiki-lint
```

### Шаг 2 — Начало каждой рабочей сессии

Открыли Agents Code или новый чат — первая команда всегда одна.

```
/wiki-start
```

Если указан аргумент (например, `/wiki-start добавить рефанд`), Agents сразу читает L1 hot cache и погружается в задачу через wiki-first.

За секунды Agents восстановит полный контекст: прочитает `wiki/hot-context.md` (если свежий), перечислит известные компоненты по `index.md`, отметит устаревшие страницы и незакрытые противоречия. Дальнейшая работа строится на этой базе — без повторных объяснений.

### Шаг 3 — Ежедневная работа

Во время разработки используйте команды по ситуации. Вопрос о проекте — `/wiki-ask`. Получили ценный ответ — скажите «сохранить», и он станет частью вики. Изменили код — `/wiki-update` (авто-детекция). Изменилась архитектура — `/wiki-repomap`. Продвинулись по задачам — `/wiki-plan`.

```
/wiki-ask как настроена обработка ошибок в платёжном модуле?
→ [Agents отвечает с цитатами]
→ «Сохранить ответ в вики» — Agents архивирует как страницу

/wiki-update          # авто-детекция: git diff или SHA-256
/wiki-repomap         # если добавились/удалились классы
/wiki-plan sprint-5: задача №3 → в работе
```

### Шаг 4 — Поддержка актуальности

Вики устаревает без ухода. Правило простое: изменил код — сразу `/wiki-update` (авто-детекция). Добавил/удалил классы — `/wiki-repomap`. Раз в спринт запускайте `/wiki-lint` — двухуровневый аудит (авто-исправления + доклад 🔴🟡🔵). Детерминированные проблемы исправит сам, остальное покажет списком с учётом lifecycle страниц.

## 6. Когда в проекте нужно вызывать скилл

Скилл активируется не только по команде. Agents обращается к `SKILL.md` автоматически при наличии указания в `Agents.md`. Понимать триггеры важно для осознанной работы с инструментом.

**Начало сессии.** Любой новый разговор о проекте должен стартовать с `/wiki-start`. Пропустив шаг, ассистент не будет знать об уже проиндексированных данных и начнёт читать исходники заново.

**После изменения источников.** Поправили конфигурацию, переписали модуль, обновили схему — запустите `/wiki-update` без аргументов. Команда сама найдёт изменившиеся файлы через git или по дате, классифицирует каждое изменение как точечное / структурное / глобальное и обновит только нужные секции. Большинство правок обновляют страницу напрямую без пометки `[устарело]`.

**При появлении нового компонента.** Добавили модуль, новый XML-сервис или пакет — занесите путь в `sources.md`, затем `/wiki-ingest` с новой меткой. Откладывать не стоит: чем раньше элемент попадёт в базу, тем быстрее команда начнёт им пользоваться.

**При обнаружении противоречия.** Нашли конфликт между поведением кода и документацией — немедленно зафиксируйте через `/wiki-update` затронутого компонента. Секция `## Противоречия` останется в вики до тех пор, пока команда не примет решение.

**При принятии архитектурного решения.** Обсудили выбор подхода, зафиксировали в ADR — сразу `/wiki-ingest adr`. Решения, остающиеся только в Confluence или голове тимлида, теряются. База знаний делает их частью постоянного рабочего контекста.

**При онбординге нового разработчика.** Достаточно показать три файла: `Agents.md`, `wiki/sources.md`, `wiki/index.md`. Новый участник запустит `/wiki-start` и получит сжатое введение в проект без часовых объяснений от коллег.

**При ревью технического долга.** Запускайте `/wiki-lint` перед ретроспективой или планированием спринта. Детерминированные проблемы исправятся сами; эвристический список станет повесткой для команды.

**Для оценки состояния вики.** `/wiki-stats` покажет сводку: количество страниц по lifecycle, покрытие источников, осиротевшие страницы, статус инфраструктуры. Полезно перед крупным релизом.

**После структурных изменений в коде.** Добавление/удаление пакетов, классов, модулей — запустите `/wiki-repomap`, чтобы обновить структурный скелет. Diff старого и нового `repo-map.md` покажет, что именно изменилось.
